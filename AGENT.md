# AI Ops Agent 项目上下文

## 项目定位

本项目建设一个企业级 AI Ops Agent 智慧运维大脑，面向 AWS 云上环境和本地私有化环境提供两套独立部署方案。两种方案共享产品能力、数据协议和核心代码，但运行时默认不互通、不共享业务数据。

项目目标不是用大模型替代全部监控工具，而是把指标、日志、链路、资源拓扑、变更记录和运维知识组织成可验证的故障处理闭环。

## 已确认决策

- AWS 云上版与本地私有化版是两套独立交付方案。
- 本地私有化版必须支持完全断网运行。
- Kubernetes 是主要运行环境，同时支持虚拟机、物理机和传统服务。
- 当前没有既定监控系统，产品需要提供默认轻量可观测套件，并保留第三方系统接入入口。
- Agent 第一阶段以只读诊断为主。
- 执行动作先经过人工审批；低风险自动执行是否启用待后续决定。
- 飞书、企业微信是通知和交互连接器，不是故障事实存储。
- 完全断网环境不能依赖公网飞书、企业微信、云模型、在线授权或在线镜像仓库。

## 第一期目标

打通以下闭环：

1. 接入告警、指标、日志和变更事件。
2. 对信号进行标准化、去重、抑制和事件聚合。
3. 建立内部事件中心保存故障事实和处理时间线。
4. 结合服务目录、运行时拓扑、变更记录和运维知识进行分析。
5. 输出带证据的根因候选、置信度、缺失信息和处理建议。
6. 通过飞书、企业微信或内网渠道通知并跟踪处理进度。
7. 保留人工审批执行入口，但第一期不默认自动修改生产环境。

## 第一期能力边界

- 异常分析：核心建设。
- 辅助根因定位：核心建设，不把模型推断包装成确定事实。
- 性能优化：先覆盖慢接口、错误率、依赖延迟和资源饱和等高价值场景。
- 安全风险：先聚合已有安全工具发现，不自行构建主动漏洞扫描器。
- 自动工单：建设内部事件记录，并与通知渠道联动。
- 自动修复：第一期不默认启用。

## 架构原则

- 规则、统计和日志聚类负责发现异常，大模型负责关联、解释和生成建议。
- 原始运维数据先在本地过滤、脱敏、聚合，再进入模型路由。
- 密钥、Token、Cookie、客户业务数据和数据库原始记录不得进入模型。
- 所有诊断结论必须保留来源、证据、时间范围和置信度。
- 服务身份必须统一，至少包含 namespace、service、environment、instance 和 version。
- 声明式服务目录、云资源关系、运行时调用拓扑和变更时间线分别维护，再在事件上下文中合并。
- 监控、模型、存储和通知都通过适配器接入，避免绑定单一厂商。
- 完全断网版本的安装、授权、升级、模型、知识库和依赖必须能够离线交付。

## 建议默认技术方向

- 采集与标准化：OpenTelemetry Collector。
- 指标：Prometheus。
- 日志：Loki，保留 OpenSearch/Elasticsearch 等适配器。
- 链路：Tempo，保留其他 OTLP 后端适配器。
- 展示：Grafana + AI Ops 自有控制台。
- 告警：Alertmanager + 外部告警 Webhook 适配器。
- 事件中心：独立服务 + PostgreSQL。
- 模型接入：统一模型网关；AWS 版优先 Bedrock，私有化版使用本地推理服务。
- 本地推理生产基线：Linux + NVIDIA GPU，具体模型通过离线评测确定。

## 开发与评审约束

- 开始实现前先阅读 `docs/README.md` 和相关专题文档。
- 未确认前不要创建 AWS 资源、购买 GPU、启用付费模型或产生持续费用。
- 未确认前不要让 Agent 执行生产变更。
- 不得声称根因已确认，除非存在可复核证据或人工确认。
- 新增组件前先说明它解决的问题，避免同时引入多套重叠的数据平台。
- 所有离线能力都要在无公网依赖的环境中验证。
- 架构或范围发生变化时，同步更新本文件和对应 `docs/` 文档。

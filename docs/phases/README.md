# 分期文档目录

本目录统一保存“智能运维大脑”各阶段的专属需求、设计、任务和验收材料。跨阶段长期有效的愿景、总体架构、安全原则、部署规范和硬件基线继续保存在 `docs/` 根目录，作为单一事实来源。

## 当前阶段

| 阶段 | 目标 | 文档入口 | 状态 |
|---|---|---|---|
| 阶段 0 | 验证 Agent Runtime、MCP、模型、存储、可观测性和完全断网链路 | [phase-0](phase-0/README.md) | 设计与技术验证准备 |
| 第一期 | 交付只读智能诊断闭环 | [phase-1](phase-1/README.md) | 需求与架构设计 |
| 第二期及以后 | 按总需求大纲逐期建立 | 尚未创建 | 未开始 |

整体路线、各期依赖和退出条件见 [需求大纲](../需求大纲.md)。

## 每期统一结构

新阶段开始时创建 `phase-{n}/`，并按实际需要维护：

```text
phase-{n}/
├── README.md                       # 本期入口、状态和阅读顺序
├── scope.md                        # 目标、范围、非目标和验收
├── requirements.md                 # 详细需求与用户流程；需要时创建
├── design.md                       # 总体设计；专题较多时拆分为独立文档
├── tasks.md                        # 任务、依赖、负责人和进度；进入实施时创建
├── test-and-evaluation-plan.md     # 测试、评测和发布门禁
├── release-and-rollback.md         # 发布、升级与回滚；交付前创建
└── decisions.md                    # 本期局部决策；需要时创建
```

允许增加领域专题文档，但每份文档必须从该期 `README.md` 可达。

## 维护规则

- 每份分期文档只描述该期增量，不复制跨期公共规则。
- 公共规则变化时更新根目录专题文档；分期文档通过链接引用。
- 新需求先写入总需求大纲并确定阶段归属，再进入对应目录。
- 状态统一使用“已确认、推荐、待验证、待确认、已实现、已延期、已取消”。
- 一期结束后保留原始范围和验收证据；后续变化进入新一期，不回写历史范围掩盖变更。
- 文档移动或改名必须同步更新 `docs/README.md`、根 `AGENTS.md` 和所有相对链接。

# 第一期范围

## 第一阶段主目标

第一期不平均实现所有 AI Ops 能力，而是交付一条可以演示、验证和持续改进的故障处理闭环：

```text
信号进入
  -> 标准化与去重
  -> 聚合为内部事件
  -> 补齐服务、拓扑和变更上下文
  -> 生成根因候选与证据
  -> 生成处理建议
  -> 通知与人工确认
  -> 记录处理进度和恢复结果
```

## P0：第一期必须完成

### 统一接入

- Prometheus/Alertmanager 告警。
- OpenTelemetry 指标、日志和 Trace。
- 通用 Webhook。
- Kubernetes 资源、事件和工作负载状态。
- 发布与配置变更事件入口。

### 事件中心

- 告警标准化和指纹。
- 去重、抑制和基础关联规则。
- 事件创建、分派、确认、处理中、恢复和关闭状态。
- 完整事件时间线。
- 原始信号、AI 结论、人工反馈和通知记录关联。

### 异常分析

- 静态阈值和动态基线入口。
- 错误率、延迟、饱和度和可用性异常摘要。
- 日志错误模式聚合。
- 相同服务、依赖和时间窗口内的信号关联。

### 辅助根因定位

每次分析至少输出：

- 受影响服务和资源。
- 根因候选列表，而不是单一武断结论。
- 每个候选对应的证据。
- 置信度和判断依据。
- 反证或缺失信息。
- 建议执行的下一条只读检查。

### 通知和处理建议

- 飞书、企业微信和通用 Webhook 连接器。
- 完全断网模式下使用内部控制台或客户内网渠道。
- 自动生成处理建议。
- 记录人工采纳、拒绝和修正结果。

### Agent Runtime 与 MCP

- 云上模型网关提供 Provider 注册与能力声明机制，支持客户配置经过批准的模型服务；开发阶段先用 DeepSeek API 和 Qwen API 完成适配和回归验证。
- 第一阶段至少支持 OpenAI-compatible 自定义 Provider，通过服务端密钥引用配置 Base URL、模型名称和 API Key。
- Air-Gapped 启动前执行主机和模型兼容性预检，未达到最低配置时失败退出。
- 实现一个模块化本地 MCP Server。
- 本地开发支持 `stdio`，Kubernetes 部署支持内部 Streamable HTTP。
- 第一批适配器覆盖事件中心、Prometheus 和 Kubernetes。
- 第一阶段使用一个诊断 Agent 和可恢复的确定性 Workflow，不直接建设复杂 Agent 网络。
- 限制单次诊断的工具数量、循环次数、运行时间和结果大小。
- 对每次工具调用记录 Agent、Scope、参数摘要、数据范围、耗时和状态。
- 使用 Mastra 验证 Workflow、MCP、暂停/恢复、PostgreSQL 存储、OpenTelemetry 和 Scorers。

## P1：第一期有限覆盖

- 慢接口与调用链热点。
- 数据库连接、慢查询摘要和缓存命中率异常。
- CPU、内存、磁盘、网络和容器资源瓶颈。
- WAF、安全平台和漏洞管理系统已有发现的聚合与影响分析。
- 人工审批后的低风险诊断动作，例如重新采集状态或执行只读查询。

## P2：后续阶段

- 主动漏洞扫描和自动渗透测试。
- 自动扩容、重启、回滚、切流和封禁。
- 复杂跨环境容量优化。
- 基于历史事件的持续异常模型训练。
- 多租户商业化计费和大规模跨组织运营。

## 第一阶段明确不做

- 不用大模型替代 Prometheus、规则引擎和时序异常算法。
- 不让模型直接 SSH 或获取集群管理员权限。
- 不同时引入 Kafka、RabbitMQ、SQS 等多套重叠消息基础设施。
- 不同时建设 OpenSearch、ClickHouse、Doris、StarRocks、Spark 和 Hive 全家桶。
- 不承诺 AI 输出天然等于真实根因。

## 验收方向

- 能从模拟或真实告警创建唯一事件。
- 重复告警不会无限生成新事件。
- 事件可以关联服务、实例、环境和最近变更。
- AI 输出包含证据和可复核查询，不只有自然语言结论。
- 飞书/企微不可用时不影响事件中心工作。
- 关闭公网后，本地版核心功能仍能运行。
- 人工可以对根因和建议进行采纳、拒绝和修正。
- 诊断 Workflow 可以在中断后恢复，不重复创建事件或重复执行已完成步骤。
- 所有 MCP 工具调用都可审计，越权、超范围和任意命令请求被拒绝。
- 完全断网验证中不产生公网 DNS、HTTP、模型或遥测请求。
- 离线硬件预检失败时不启动模型和 Agent Runtime，也不自动回退到云端或低规格模式。

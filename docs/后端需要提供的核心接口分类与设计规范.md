为了配合前述以 **React + Vercel AI SDK + React Flow** 打造的智能运维大脑前端，后端（采用 **Go** 或 **Hono** 等服务）需要提供一套涵盖 **实时流式交互、可观测性数据透传、拓扑计算与人机安全闭环** 的标准 API 接口体系。

以下是后端需要提供的核心接口分类与设计规范：

---

## 1. 🤖 AI Agent 核心交互接口 (Agent & Streaming API)

这是右侧 Copilot 区和推理引擎通信的核心，建议使用 **SSE (Server-Sent Events)** 实现流式推送。

### 1.1 发起诊断与流式对话

* **`POST /api/v1/agent/chat`** 或 **`GET /api/v1/agent/stream`**
* **作用**：处理前端发起的对话请求或针对特定 Incident 的分析。
* **协议**：`text/event-stream` (SSE)
* **后端输出事件流**（推送机制）：
```json
// Event: reasoning (思维链推演)
data: {"type": "reasoning", "content": "正在从 Prometheus 拉取过去 15 分钟的 CPU 波动..."}

// Event: tool_call (工具调用状态)
data: {"type": "tool_call", "tool": "query_sentry", "status": "running", "params": {"issue_id": "1042"}}

// Event: text (流式生成文本)
data: {"type": "text", "delta": "根因分析完成，系 Redis 内存溢出导致。"}

// Event: action_request (触发 Human-in-the-loop 确认卡片)
data: {"type": "action_request", "action_id": "act_8891", "type": "restart_pod", "risk": "HIGH", "target": "orders-v2"}

```





### 1.2 人机协作决策授权 (Human-in-the-Loop)

* **`POST /api/v1/agent/actions/:action_id/confirm`**
* **作用**：前端点击高风险卡片上的“确认授权执行”或“拒绝”时调用。
* **Request Body**：`{ "approved": true, "reason": "已人工核对，允许重启" }`


* **`POST /api/v1/agent/actions/:action_id/cancel`**
* **作用**：中断 Agent 正在执行的长流程任务。

## 2. 🚨 告警聚合与根因分析接口 (Incident & RCA API)

用于中间主工作区渲染告警列表、压缩率统计以及 React Flow 根因拓扑。

### 2.1 聚合事件列表 (Incidents)

* **`GET /api/v1/incidents`**
* **作用**：获取经过 AI 收敛后的 Incident 列表（而非原始的数千条 Alertmanager 垃圾信息）。
* **Query 参数**：`status=active&severity=critical&limit=20`
* **Response**：
```json
{
  "incidents": [
    {
      "id": "inc_20260721_001",
      "title": "SQS 消息消费延迟与前端 500 异常",
      "noise_reduction_rate": "88%", // 压缩率
      "raw_alert_count": 142,        // 聚合的原始告警数
      "root_cause_summary": "Redis 连接池满导致线程阻塞",
      "status": "ANALYZING"
    }
  ]
}

```





### 2.2 根因拓扑图数据 (RCA Graph Data)

* **`GET /api/v1/incidents/:id/topology`**
* **作用**：直接返回可供 **React Flow** 渲染的 Node 与 Edge 数据。
* **Response**：
```json
{
  "nodes": [
    { "id": "node-1", "type": "sentry", "label": "前端 Error (500)", "status": "error" },
    { "id": "node-2", "type": "sqs", "label": "SQS 死信队列 (DLQ)", "status": "warning" },
    { "id": "node-3", "type": "redis", "label": "Redis 连接池 (根因)", "status": "critical", "is_root_cause": true }
  ],
  "edges": [
    { "id": "e1-2", "source": "node-1", "target": "node-2", "animated": true },
    { "id": "e2-3", "source": "node-2", "target": "node-3", "animated": true }
  ]
}

```

## 3. 📊 可观测性数据与下钻接口 (Telemetry API)

前端点击任意节点或证据时，需要快速调取底层的细粒度指标与日志。

### 3.1 综合态势与数据源健康度

* **`GET /api/v1/telemetry/health`**
* **作用**：返回顶部的系统健康评分（Score）以及左侧 Sentry、Prometheus、Fluentd、AWS SQS 的连通状态。



### 3.2 证据下钻 (Evidence Drill-down)

* **`GET /api/v1/telemetry/metrics/query`**：向 Prometheus / CloudWatch 发起 PromQL / 时序数据查询，供 Recharts 渲染。
* **`GET /api/v1/telemetry/logs/search`**：按 `trace_id` 或 `incident_id` 查询 Fluentd / OpenSearch 日志。
* **`GET /api/v1/telemetry/sentry/errors/:error_id`**：获取 Sentry 的真实 Error StackTrace 堆栈。

## 4. 🛠 自愈动作与工单系统接口 (Ops Operations API)

当运维工程师点击“一键修复”或“生成工单”时调用的后端接口。

### 4.1 执行自愈/运维指令

* **`POST /api/v1/ops/remediate`**
* **作用**：触发特定的运维 Playbook 或云服务 API（如 SQS 死信队列消息重刷 Redrive、Pod 重启、HPA 动态扩容）。
* **Request Body**：`{ "incident_id": "inc_001", "action_type": "sqs_redrive", "params": { "queue": "orders-dlq" } }`



### 4.2 工单与通知集成

* **`POST /api/v1/tickets/create`**
* **作用**：自动将当前 Incident 的分析报告、日志证据和 RCA 拓扑一键导出，并在 JIRA、飞书或钉钉中创建工单。



---

## 💡 后端接口设计核心要点总结

| 维度 | 建议设计策略 |
| --- | --- |
| **实时性方案** | AI 对话与推理优先选用 **SSE** (Server-Sent Events)；如果是全屏告警大屏实时刷新，可配合 **WebSocket**。 |
| **前端适配优化** | 根因拓扑接口可以直接将节点格式转换为 React Flow 识别的 `{ nodes, edges }` JSON 结构，减少前端转化逻辑。 |
| **幂等与安全** | `POST /api/v1/ops/remediate` 等高危动作接口**必须带有 Idempotency Key（幂等键）**，防止 AI 或用户重复点击导致多次重启服务。 |
